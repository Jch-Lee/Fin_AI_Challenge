### 1 목적 및 범위
#### 목적
정보주체가 정보전송자에게 제3자전송요구를 행사하는 절차를 안내하고, 참여 기관 별 안전하고 효율적으로 전송요구권을 처리하는 절차를 안내함에 있다.
#### 범위

# 01. 가이드라인 개요
## 3. 용어 정의
### 개인 정보
살아 있는 개인에 관한 정보로서 성명, 주민등록번호 및 다른 정보와 쉽게 결합하여 알아볼 수 있는 정보로 「개인정보 보호법」제2조제1호 상의 정의를 따름
### 정보 주체
처리된 개인정보에 의하여 알아볼 수 있는 정보의 주체가 되는 사람으로서 전송요구권을 행사하는 자
### 전송요구권
정보주체가 개인정보처리자에게 자신의 정보를 본인 또는 본인이 지정한 제3자에게 전송할 것을 요구할 수 있는 권리 (「개인정보 보호법」제35조의2)
※본 가이드라인에서는 별도 명시가 없는 경우 제3자에게 전송하는 권리를 말함
### 전송요구 대상 정보
정보주체가 전송요구권을 행사할 수 있는 개인정보로서 법으로 정한 기준에 해당하는 정보 (「개인정보 보호법」제35조의2제1항제1호 내지 제3호)
### 정보 전송자
정보주체의 전송요구에 따라 보유한 개인정보를 정보주체 본인 또는 정보수신자에게 전송해야 하는 자
### 정보 수신자
정보주체의 전송요구를 통해 정보를 수신받아 아래와 같은 업무를 수행하는 자
1) 일반 수신자
개인정보를 전송받아 확인하는 자 (「개인정보 보호법 시행령」제42조의3)
2) 일반 전문기관
제3자 대상 정보 전송자로부터 전송받은 개인정보(보건의료 정보 제외)를 관리·분석하여 통합조회, 맞춤형 서비스, 연구, 교육 등에 이용하는 업무를 수행하는 자 (「개인정보 보호법 시행령」제42조의9제1항제2호)
3) 특수 전문기관
제3자 대상 정보 전송자로부터 전송받은 개인정보 중 보건의료 정보를 관리·분석하여 통합조회, 맞춤형 서비스, 연구, 교육 등에 이용하는 업무를 수행하는 자 (「개인정보 보호법 시행령」제42조의9제1항제3호)
### 중계 전문기관
정보주체의 전송요구에 따라 정보 전송자의 전송을 지원하는 업무를 수행하는 자 (「개인정보 보호법 시행령」제42조의9제1항제1호)
### 참여 기관
정보주체의 전송요구권 보장을 위해 참여하는 정보 전송자, 정보 수신자, 중계 전문기관, 인증 기관, 개인정보 전송 지원 플랫폼 등을 총칭

#### 전송요구서

#### 전송요구
- 정보주체가 정보수신자를 통해 전송요구서를 중계전문기관에게 전송하고, 중계전문기관은 전송요구서의 정당성을 확인하여 정보수신자에게 정보 요청 권한(접근토큰)을 발급하는 과정
(「개인정보 보호법 시행령」 제42조의5)
#### 전송요구 유형

3. 주민등록번호 기반 유형 : 주민등록번호를 연계정보로 변환 후 정보주체를 식별하여 전송요구

### 텍스트
#### 정보 요청
전송요구로 발급받은 정보 요청 권한(접근토큰)을 이용하여 정보수신자가 정보전송자에게 정보를 요청하고, 정보전송자가 정보수신자에게 정보를 전송하는 절차
#### API (Application Programming Interface)
참여기관 간에 정보를 송수신하기 위한 미리 정의된 표준화된 전송규격 및 절차
1. 전송요구 API
정보수신자가 전송요구서를 중계전문기관에게 전송하여 접근토큰을 발급받는 절차의 API
2. 정보요청 API
정보수신자가 접근토큰을 이용하여 개인정보를 전송할 수 있도록 중계전문기관이 제공하는 API
3. 전송지원 API
참여기관 간 상호 연결에 필요한 정보를 제공하거나, 정보를 송수신하기 위한 API
#### 자격증명 (Credential)
API 요청 시 참여기관 간 자격을 인증하고 식별하기 위해 개인정보 전송지원 플랫폼으로 부터 발급받는 정보
#### 본인식별 검증
정보주체의 동일인 여부를 확인하기 위하여 각각의 연계정보 일치 여부를 확인하는 절차
1. 정보수신자 연계정보
정보수신자가 정보주체 회원가입 시 본인확인을 통해 확보한 연계정보
2. 정보전송자 연계정보
정보전송자가 정보주체 회원가입 시 본인확인 또는 「정보통신망법」제23조의5에 따른 연계정보 변환을 통해 확보한 연계정보
3. 인증기관 연계정보
인증기관의 전자서명 인증결과 및 위임검증 결과에 포함된 연계정보

### 1 참여기관별 역할
#### 1.1 정보주체
- 정보주체는 개인정보 전송지원 플랫폼을 통해 정보수신자에게 정보를 전송합니다.
#### 1.2 정보수신자
- 정보수신자는 정보주체로부터 정보를 받고, 중계전문기관을 통해 정보를 전송합니다.
#### 1.3 중계전문기관
- 중계전문기관은 정보수신자와 정보전송자 간의 정보 전송을 중개하며, 정보전송자의 정보를 확인하고 전송합니다.
#### 1.4 정보전송자
- 정보전송자는 정보수신자에게 정보를 전송하며, 정보수신자와 중계전문기관을 통해 정보를 전송합니다.
#### 1.5 인증기관
- 인증기관은 정보전송자의 자격증명을 확인합니다.
### 제3자 대상 정보전송자(이하 ‘정보전송자’)
- 정보전송자는 정보주체(정보수신자)의 요구에 따라 정보를 추출하고, 중계전문기관을 통해 전송의 안정성·신뢰성이 보장되도록 개인정보보호위원회(이하 ‘보호위원회’)가 고시하는 방식으로 정보를 전송해야 합니다.
- 정보전송자는 이를 위해 정보주체의 전송요구 조건에 따른 정보추출 및 중계전문기관 연동 시스템 등을 개발해야 한다.
### 정보수신자(일반수신자, 일반전문기관, 특수전문기관)
- 정보수신자는 정보주체의 전송요구를 중계전문기관에게 전달하여 정보요청 권한을 부여받고, 이를 이용하여 정보를 전송받을 수 있다.
- 따라서 정보수신자는 정보주체의 전송요구를 받아 전송요구서를 생성하고, 중계전문기관 연동 및 마이데이터 서비스를 제공하기 위한 기능의 개발이 필요하다.

# 5.2.① 개별인증
- 정보주체는 정보전송자의 인증 화면을 통해 개별인증을 수행한다.
- 정보수신자는 정보주체에게 개별식별자 유형의 정보전송자 선택 화면 제공
- 정보주체는 전송요구 대상이 되는 정보전송자 선택
- 정보수신자는 정보전송자가 제공하는 ‘개별인증 인가코드 발급 API(API 전송요구-004)’를 호출하고 정보전송자의 인증 화면을 정보주체에게 제공
- 정보전송자는 정보주체가 입력한 인증정보 검증
## 정보전송자 개별인증 수단 제공 지침
- (개별인증 수단을 제공해야 하는 자) 정보주체의 연계정보를 보관하지 않거나, 일부만 보관하는 정보 전송자
- (제공 형태) 정보전송자가 정보주체 회원가입 시 확보한 회원계정 및 비밀번호, 휴대폰번호, PIN번호를 이용하거나, 전용인증앱, 소셜 로그인 등 정보주체를 인증할 수 있는 수단으로 정보수신자의 웹 또는 앱에서 적용할 수 있는 방식
- 개별인증 수단을 제공하는 정보전송자는 인증수단을 호출할 수 있는 URL 또는 앱 스킴을 마이 데이터 플랫폼에 사전 등록해야 함

① 표준 연결 방식
- HTTP을 기반으로 참여기관 간 안전한 통신을 위해 TLS기반 상호인증(이하 mTLS)을 전송 구간 암호화에 적용한다.
**mTLS (Mutual Transport Layer Security)**
- 전송(Transport) 계층과 응용(Application) 계층 사이에서 종단 간 인증, 전송 데이터의 암호화, 무결성을 보장하는 표준 프로토콜
- API를 호출하는 클라이언트와 API 제공자 간 양방향 인증으로 신뢰성 있는 통신을 지원
- **TLS 암호화 통신 버전은 1.3 이상 사용**
- TLS 1.2는 HEIST, DWOWN, FREAK, POODLE 공격 및 HeartBleed 취약점 등이 보고
- 「표준 API 규격 및 명세」 상에 mTLS 예외로 표기된 경우를 제외하고 모든 연결은 mTLS를 적용
- 예) 정보수신자 서비스와 정보전송자가 직접 통신해야 하는 ‘개별인증 API’의 경우는 예외
##### 2 SSL/TLS 인증서 적용
- 참여기관 간 상호인증, 암호화된 정보의 송 · 수신을 위해 인증기관(CA, Certificate Authority)으로부터 OV등급 또는 EV등급 SSL/TLS 인증서를 발급받아 적용한다.
**OV(Organization Validated) 등급 TLS 인증서**
- 인증기관(CA)이 도메인 소유권과 조직의 실제 여부를 검증한 뒤 발급되는 인증서
- 와일드카드 도메인에 대한 인증서 발급이 가능(예시: *.domain.com)
**EV(Extended Validation) 등급 TLS 인증서**
- 도메인 소유권과 조직의 실제 여부, 법적 존재 및 운영 여부를 인증기관(CA)이 검증 후 발급하는 인증서

- 인증서 시리얼 번호에 사업자 번호 혹은 법인 번호를 명시하여 발급 가능

# 1.1.2 데이터 표현 규격
## ① 데이터 표현 형식
API 요청 · 응답 메시지 형식은 JSON을 사용하며, 분야별 · 산업별로 사용하는 표현방식을 적용한 경우는 예외로 한다.
### JSON(JavaScript Object Notation)
- 속성 · 값(Key:Value)의 집합으로 이루어진 데이터 객체를 표현하기 위한 개방형 표준 메시지 형식
- 국제표준(ECMA-404)로 채택되어 있으며, 웹서비스에서의 사실상 표준 데이터 형식으로 사용됨
## ② 데이터 인코딩
메시지 전송을 위한 인코딩 방식은 UTF-8을 사용한다.
### UTF-8
- ASCII 코드를 확장하여 전 세계의 모든 문자코드를 표현할 수 있는 표준 인코딩 방식으로써 범용성이 높아 호환성이 우수
- 웹 서비스 표준으로 ASCII와 호환되어 다양한 시스템에서 지원
## ③ 데이터 길이

### UUID v7(Universally Unique Identifier Version 7)
- 시간 기반(타임스탬프 기반)으로 고유한 식별자를 생성하는 방식
- 시계열 값의 이점인 삽입 정렬, 데이터베이스 인덱싱에서 유리
- 동일한 시간에 생성된 UUID도 고유성 보장
- UUID생성 시 MAC주소나 서버이름과 같은 식별정보를 포함하지 않아 보안성 우수
- IETF에 의해 지정된 표준으로 RFC 9562에 명세(예시 : 018f8401-55ac-7ba4-b3f6-45ca5fa453e4)
### ⑦ 응답 코드 및 메시지
표준 HTTP 응답 코드만으로는 다양한 상태를 표현하는데 한계가 있으므로, 상황별 세부 응답 코드 및 메시지를 별도로 정의하여 사용한다.
- API 요청자(API를 호출하는 측)의 오류는 400번대로 응답
- 접근토큰 발급 API(OAuth 준용)는 RFC 6749 및 7009 국제표준준용
- API 응답자(API호출에 응답하는 측)의 오류는 500번대로 응답
- 그 외 응답은 'rsp_code(응답 코드)'와 'rsp_msg(응답 메시지)' 이용
### ⑧ 페이지네이션
이력정보 등 한 번의 응답으로 전송이 불가능한 경우는 부분 범위 조회 기법(Cursor-Based Pagination)을 적용하여, 응답 메시지 당 전송하는 데이터 크기를 조절하여 응답할 수 있다.
#### Cursor-Based Pagination
- 대량의 데이터를 페이지라는 단위로 나누어 처리하는 방식으로 페이지를 탐색하여 효율적인 메시지 반환이 가능
- 웹 서비스와 모바일 어플리케이션에서 데이터를 동적으로 로드하거나 표시할 때 주로 사용
| 구분 | 이름 | 타입 (길이) | 설명 |
|------|------|-------------|-------|
| 요청 | limit | Integer(3) | 목록 내 반환될 개체의 개수 (최대 500까지 설정 가능) |
|      | next_page | String(1000) | 다음 페이지 요청을 위한 기준개체 (설정 시 해당 개체를 포함한 limit 개 반환) |
| 응답 | next_page | String(1000) | 다음 페이지 요청을 위한 기준개체 (다음 페이지 존재하지 않는 경우 미회신) |
- 기준개체 식별자의 생성규칙은 정보전송자가 자율적으로 정함. 다만, 특수문자를 포함하여 기준개체 식별자를 생성하는 정보전송자의 경우 URL safe한 방식(URL encoding, URL Safe BASE64 등)을 적용하여 생성 및 응답
- 단, limit을 최대 500까지 설정하더라도 정보전송자의 환경에 따라 페이지 당 반환되는 개체의 개수는 변경 가능(0 ≤ 반환되는 개체 개수 ≤ limit)

- IETF RFC 7515 – JSON Web Signature(JWS)
- JWS (JSON Web Signature)
- JWT에 전자서명을 결합한 형태로 토큰의 무결성과 메시지 인증을 제공하며, OAuth 2.0에서 보안을 강화할 목적으로 JWS를 사용

- JWS 규격 토큰은 Header, Payload, Signature로 구성되며, BASE64로 인코딩한 후 마침표로 연결한 형태로 생성한다.
